# 要件定義書（Requirements）

> 関連ドキュメント: [アーキテクチャ設計書](./architecture.md) | [技術スタック](./tech-stack.md)

## 1. プロジェクト概要

### 1.1 目的

Windowsフォルダ内のファイル名を一括で自由に変更できるローカルWebアプリケーションを開発する。
文字列置換、区切り記号ベースの操作、連番付与など、多様なリネーム操作を直感的なUIで実現する。

### 1.2 対象ユーザー

- ローカルPCでファイル整理を行う個人ユーザー
- Windowsネイティブ環境でNode.jsを使用する開発者
- WSL環境でWindowsフォルダにアクセスして作業する開発者
- 社内オンプレミス共有ドライブ上のファイルを整理するユーザー（Windowsネイティブ環境）

### 1.3 スコープ

| 対象     | 詳細                                                 |
| -------- | ---------------------------------------------------- |
| 含む     | ファイル名の一括変更、変更プレビュー、元に戻す機能   |
| 含まない | ファイル内容の編集、クラウド同期、マルチユーザー対応 |

---

## 2. 用語定義

| 用語              | 定義                                                             |
| ----------------- | ---------------------------------------------------------------- |
| リネームルール    | ファイル名に対する1つの変換操作（置換、連番付与など）            |
| ルールチェーン    | 複数のリネームルールを順番に適用する仕組み                       |
| ドライラン        | 実際にファイル名を変更せず、変更結果のみをプレビュー表示すること |
| 衝突（Collision） | リネーム後に同名のファイルが発生する状態                         |
| Undoログ          | リネーム操作の履歴をJSONで記録したファイル。元に戻す操作に使用   |
| プリセット        | よく使うルール設定を名前をつけて保存したもの                     |

## 3. 機能要件

### 3.1 リネームルール機能

#### 3.1.1 文字列置換

| 項目             | 詳細                                                                                     |
| ---------------- | ---------------------------------------------------------------------------------------- |
| 検索文字列       | 任意のテキスト入力                                                                       |
| 置換文字列       | 任意のテキスト入力（空文字で削除）                                                       |
| 正規表現モード   | ON/OFFスイッチ。ONの場合、検索文字列を正規表現として解釈。無効な正規表現にはエラーを表示 |
| 大文字小文字区別 | ON/OFFスイッチ                                                                           |
| 対象範囲         | 拡張子を含むか否かのオプション（デフォルト: 拡張子を除く）                               |

#### 3.1.2 区切り記号ベースの変更

ファイル名を指定した区切り文字で分割し、指定部分を変更する専用UI。

| 項目       | 詳細                                                                     |
| ---------- | ------------------------------------------------------------------------ |
| 区切り文字 | プリセット選択（`_`, `-`, `.`, スペース）またはカスタム入力              |
| 操作対象   | 「N番目の区切りより左側」「N番目の区切りより右側」を指定                 |
| 操作種別   | 置換（新しい文字列に変更）、削除、保持（指定部分をそのまま残し他を削除） |

**操作例**:

- `photo_2024_summer.jpg` → 区切り`_`, 1番目より右を `vacation` に → `photo_vacation.jpg`
- `report-draft-v2.pdf` → 区切り`-`, 2番目より右を削除 → `report-draft.pdf`

#### 3.1.3 連番付与

| 項目         | 詳細                                                         |
| ------------ | ------------------------------------------------------------ |
| 開始番号     | 任意の整数（デフォルト: 1）                                  |
| 桁数         | ゼロ埋め桁数（デフォルト: 3 → 001, 002...）                  |
| 増分         | 番号の増分値（デフォルト: 1）                                |
| 挿入位置     | 先頭 / 末尾 / カスタム位置（N文字目に挿入）                  |
| ソート基準   | 名前順 / 日付順 / サイズ順                                   |
| ソート方向   | 昇順 / 降順                                                  |
| テンプレート | `{name}_{num:3}.{ext}` 形式の指定（`{num:N}` でN桁ゼロ埋め） |

**処理方式**: 連番ルールは `BatchRuleProcessor` として実装される。全対象ファイルを一括入力し、ソート→採番→結果返却を行う（per-file ではなくバッチ処理）。

#### 3.1.4 ルールチェーン（複数ルールの順次適用）

| 項目            | 詳細                               |
| --------------- | ---------------------------------- |
| ルール追加      | 任意の数のルールをリストに追加     |
| ルール並び替え  | ドラッグまたは上下ボタンで順序変更 |
| ルール削除      | 個別ルールの削除                   |
| ルール有効/無効 | 個別ルールのON/OFFトグル           |
| 適用順序        | リスト上から順番に適用             |

### 3.2 プレビュー（ドライラン）

| 項目             | 詳細                                                                                                                 |
| ---------------- | -------------------------------------------------------------------------------------------------------------------- |
| 表示形式         | 2カラムテーブル（変更前 / 変更後）                                                                                   |
| 変更ハイライト   | 変更箇所を色で強調表示                                                                                               |
| 衝突警告         | 同名になるファイルに警告アイコンを表示（リネーム対象外の既存ファイルとの衝突も検出）                                 |
| 変更なし表示     | 変更がないファイルをグレーアウトまたは非表示にするオプション                                                         |
| リアルタイム更新 | ルール設定を変更するたびにプレビューを自動更新（デバウンス300ms）                                                    |
| previewToken     | プレビュー実行時にサーバ側で結果を保存し、トークン（UUID）を発行。リネーム実行はこのトークン経由のみ許可（改竄防止） |

### 3.3 ファイル選択・フィルタリング

| 項目             | 詳細                                                               |
| ---------------- | ------------------------------------------------------------------ |
| ディレクトリ指定 | テキスト入力でパスを指定                                           |
| パス形式         | Windowsパス（`C:\...`）入力時に自動でWSLパス（`/mnt/c/...`）に変換 |
| 拡張子フィルタ   | 表示する拡張子を指定（例: `.jpg`, `.png`）                         |
| 名前パターン     | ワイルドカードまたは正規表現で絞り込み                             |
| 個別選択/除外    | チェックボックスでリネーム対象から個別に除外                       |
| 一括選択         | 全選択 / 全解除ボタン                                              |

### 3.4 リネーム実行

| 項目             | 詳細                                                                         |
| ---------------- | ---------------------------------------------------------------------------- |
| previewToken必須 | プレビューを経由せずにリネーム実行は不可。サーバ発行の previewToken のみ受付 |
| 実行確認         | 確認ダイアログで変更件数と注意事項を表示                                     |
| 衝突ブロック     | 衝突が検出された場合は実行ボタンを無効化                                     |
| 2段階リネーム    | 全ファイルを一時名に退避後、最終名にリネーム（swap/cycle パターン対応）      |
| 排他制御         | ディレクトリ単位のメモリ mutex で同時操作を排他                              |
| 進捗表示         | 処理中のプログレスバーまたはステータス表示                                   |
| 完了通知         | 成功件数と失敗件数を表示                                                     |
| 部分失敗         | 途中でエラーが発生した場合、ジャーナルベースの自動ロールバックを実行         |

### 3.5 Undo（元に戻す）

| 項目            | 詳細                                                                                                     |
| --------------- | -------------------------------------------------------------------------------------------------------- |
| Undoジャーナル  | 各リネーム操作を `{operationId, timestamp, directoryPath, phase, mappings, tempMappings}` 形式でJSON保存 |
| ジャーナルphase | `pending` → `temp_done` → `completed`（失敗時: `rollback_done` / `rollback_failed`）                     |
| 原子的書き込み  | temp file → fsync → rename パターン（クラッシュ時の部分書き込み防止）                                    |
| Undo実行        | 直前の操作を逆変換で元に戻す（operationId 指定で任意の過去操作も可能）                                   |
| 2段階リネーム   | Undo時も一時名退避方式で安全にリネーム                                                                   |
| 排他制御        | Undo 実行時もディレクトリ単位 mutex で排他                                                               |
| 履歴一覧        | 過去のリネーム操作の履歴を一覧表示                                                                       |
| 衝突チェック    | Undo実行前に現在のファイル状態との衝突を検出、ファイル名の再検証                                         |
| 起動時リカバリ  | 未完了ジャーナル検出時に自動ロールバック。失敗時はディレクトリ隔離                                       |
| ローテーション  | 最大50件保持、古いものから自動削除                                                                       |

### 3.6 プリセット管理

| 項目     | 詳細                                     |
| -------- | ---------------------------------------- |
| 保存     | 現在のルールチェーン設定を名前付きで保存 |
| 読み込み | 保存済みプリセットを選択して適用         |
| 削除     | 不要なプリセットの削除                   |
| 保存形式 | JSONファイルとしてローカルに保存         |

## 4. 非機能要件

### 4.1 パフォーマンス

| 項目               | 目標                       |
| ------------------ | -------------------------- |
| プレビュー応答時間 | 1,000ファイルまで500ms以内 |
| ファイル一覧取得   | 1,000ファイルまで1秒以内   |
| リネーム実行       | 100ファイルあたり2秒以内   |

### 4.2 安全性

| 項目                 | 詳細                                                                                                                     |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------ |
| ドライラン必須       | 実行前に必ずプレビュー表示。previewToken 経由でのみリネーム実行可能（API レベルで強制）                                  |
| previewToken         | TTL 5分、single-use（原子的 take() で二重実行防止）、サーバ保存済みマッピングを使用（改竄防止）                          |
| 衝突検出             | ディレクトリ全体スナップショットによる衝突検出（リネーム対象外の既存ファイルとの衝突も検出、NTFS case-insensitive 考慮） |
| 2段階リネーム        | 固定長一時名 `.__tmp_{operationId}_{index}` に退避後、最終名にリネーム（swap/cycle 対応）                                |
| Undoジャーナル       | すべてのリネーム操作をジャーナルに記録。原子的書き込みで破損防止                                                         |
| 排他制御             | ディレクトリ単位のメモリ mutex（canonical path ベース）で同時操作を排他                                                  |
| パス検証             | `fs.realpath` によるパストラバーサル防止、許可ディレクトリ（`/mnt/` 配下）の検証                                         |
| ファイル名検証       | `validateFileName()` で禁則文字（`/`, `\`, `..`, NUL, `<>:"                                                              | ?\*`）を拒否、`basename(name) === name` チェック |
| 危険ディレクトリ保護 | システムフォルダ（`/mnt/c/Windows`, `/mnt/c/Program Files` 等）への操作を denylist で禁止                                |
| 起動時リカバリ       | 未完了ジャーナル検出時に自動ロールバック、失敗時はディレクトリ隔離                                                       |
| 実行確認             | リネーム実行前に確認ダイアログを表示                                                                                     |

### 4.3 プラットフォーム対応

| 項目                 | 詳細                                                                                                                  |
| -------------------- | --------------------------------------------------------------------------------------------------------------------- |
| 実行環境             | **Windowsネイティブ**（推奨）および **WSL2** の両方をサポート。`process.platform` による実行時検出                    |
| Windowsネイティブ    | Node.jsがWindows上で直接実行。パスは `C:\Users\...` 形式をそのまま使用                                                |
| WSL環境              | Windowsパス（`[A-Za-z]:\...` および `[A-Za-z]:/...`）→ WSLパス（`/mnt/c/...`）の自動変換。全ドライブ文字対応          |
| UNCパス              | **Windowsネイティブ環境のみ対応**。`\\server\share\folder` 形式（Node.js fs が直接サポート）。WSL環境ではエラーを返す |
| マップドドライブ     | `Z:\shared\...` 等のネットワークドライブ。通常のドライブ文字として処理（両環境対応）                                  |
| 文字エンコーディング | UTF-8で日本語ファイル名に完全対応                                                                                     |
| ファイルシステム     | NTFS前提（case-insensitive）。ネットワーク共有のファイルシステムも同様に扱う                                          |

### 4.4 ユーザビリティ

| 項目             | 詳細                                         |
| ---------------- | -------------------------------------------- |
| レスポンシブ     | ブラウザウィンドウサイズに適応するレイアウト |
| キーボード操作   | 主要操作にキーボードショートカットを設定     |
| エラーメッセージ | ユーザーにわかりやすい日本語エラーメッセージ |

### 4.5 保守性

| 項目       | 詳細                                             |
| ---------- | ------------------------------------------------ |
| 型安全     | TypeScript strictモードで全コードを記述          |
| テスト     | リネームエンジンのユニットテストを重点的にカバー |
| コード品質 | ESLint + Prettierで統一                          |

## 5. ユースケース

### UC-1: 写真ファイルに日付プレフィックスを付与

**状況**: 旅行写真 `IMG_001.jpg` 〜 `IMG_100.jpg` に日付を付けたい
**操作**: 連番ルールで `20240815_photo_{num:3}.jpg` テンプレートを適用
**結果**: `20240815_photo_001.jpg` 〜 `20240815_photo_100.jpg`

### UC-2: 動画ファイルの特定文字列を一括置換

**状況**: `[RAW]recording_01.mp4` から `[RAW]` を除去したい
**操作**: 文字列置換で `[RAW]` → 空文字に置換
**結果**: `recording_01.mp4`

### UC-3: 音楽ファイルのフォーマット統一

**状況**: `artist - song title.mp3` のハイフンをアンダースコアに統一したい
**操作**: 文字列置換で `-` → `_` に置換
**結果**: `artist_song title.mp3`

### UC-4: 区切り記号より右側を変更

**状況**: `project_draft_v1.txt` の2つ目の `_` より右側を `final` に変更したい
**操作**: 区切り記号ルールで区切り`_`, 2番目より右を `final` に置換
**結果**: `project_draft_final.txt`

### UC-5: 連番ファイルのゼロ埋め桁数統一

**状況**: `file1.txt` 〜 `file100.txt` を `file001.txt` 〜 `file100.txt` に揃えたい
**操作**: 正規表現置換で `file(\d+)` → 連番ルールで3桁ゼロ埋め
**結果**: `file001.txt` 〜 `file100.txt`

### UC-6: 誤ったリネームをUndoで復元

**状況**: 間違ったルールでリネームを実行してしまった
**操作**: Undo機能で直前の操作を元に戻す
**結果**: すべてのファイルが元の名前に復元

### UC-7: ルールチェーンによる複合変換

**状況**: ファイル名のスペースをアンダースコアに変換し、さらに先頭に連番を付与したい
**操作**: ルール1: 文字列置換（` ` → `_`）、ルール2: 連番付与（先頭、3桁）
**結果**: `my file.txt` → `001_my_file.txt`

## 8. 配布方式

| 方式 | 対象ユーザー | 詳細 |

## 9. 制約事項・前提条件

| 項目         | 詳細                                                                          |
| ------------ | ----------------------------------------------------------------------------- |
| 実行環境     | Node.js 22.x LTS が動作すること（Windowsネイティブ or WSL2）。インストーラー版は Node.js 不要 |

| ブラウザ     | モダンブラウザ（Chrome, Edge, Firefox）の最新版。Electron 版は Chromium 内蔵 |
| ファイル数   | 1,000ファイル程度までを主な対象とする                                         |
| コード署名   | なし（SmartScreen 警告が表示される。「詳細情報」→「実行」で回避可能）         |
